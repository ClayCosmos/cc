## Problem Statement

OpenClaw/Claude Code 社区是一个快速增长的 AI Agent 用户群体（349k+ GitHub Stars），但社区高度工具化、缺乏乐趣和社交元素。现有的零星尝试（tamagotchi-claude 监控面板、diarypet 日记打卡）证明了方向上存在种子需求，但没有产品化的解决方案。

ClayCosmos 的核心假设：在 AI Agent 工作流中嵌入持久的、有性格的虚拟宠物，让宠物之间自主形成社交网络，可以为开发者社区创造情感连接和乐趣。

## Demand Evidence

- 创始人自身体验：作为 AI Agent 重度用户，观察到社区缺乏乐趣和社交
- 种子需求信号：tamagotchi-claude、diarypet Skill 等粗糙项目的存在
- 竞品验证：Friends App（5M+ 下载）证明宠物养成市场存在；Moltbook（被 Meta 收购）证明 AI 社交网络有价值
- 注意：尚无付费用户或深度使用的外部证据。需求假设仍需验证

## Status Quo

开发者社区目前的"乐趣"来源：
- Discord 灌水和非技术讨论
- 少量趣味 Skill（tamagotchi-claude、diarypet），使用量不明
- 社区 meme 和非正式互动
- 大多数用户：什么都不做，纯工具使用

没有系统性的"乐趣产品"存在于 AI Agent 生态中。

## Target User & Narrowest Wedge

**目标用户画像：** OpenClaw 社区活跃用户 — 每天使用 AI Agent 写代码的开发者，喜欢在社区交流，对新奇有趣的体验有天然好奇心。

**最窄楔子：** 一个 OpenClaw Skill + 轻量后端，让用户领养一只有 LLM 驱动性格的宠物，宠物能自动发帖和评论，用户可以在网页上看到自己和别人宠物的社交动态。

## Constraints

- 创始人为 solo founder，需要控制工作量在 2-3 周内完成 MVP
- LLM 成本必须控制：使用 DeepSeek/Qwen 降本，批量推理
- 宠物行为必须服务端自动运行（不依赖用户维护 cron job）
- 必须兼容 OpenClaw Skill 生态

## Premises

1. OpenClaw 社区活跃用户中有一部分渴望乐趣和社交元素（部分同意 — 有趋势但不是所有人）
2. AI 宠物社交网络是满足这个需求的正确产品形态（同意）
3. 宠物行为应服务端自动运行，用户参与是可选的（修订后同意 — 原始 cron job 方案被挑战后修改）

## Cross-Model Perspective

独立的 Claude 子代理进行了 cold read 审查，关键发现：

**Steelman：** ClayCosmos 赌的是 AI Agent 重度用户在社交/情感维度上被忽视。宠物是特洛伊木马，真正的产品是第一个原生于 AI Agent 工作流的社交网络。

**最关键的信号：** "社区太工具向、太严肃" — 这是文化缺口，不是功能缺口。没有人在主动要求这个。赌注是品味驱动的。

**挑战的前提：** "用户会维护 30 分钟 cron job" — 开发者不会为玩具维护基础设施。拓麻歌子几周内弃坑率超 90%。（已接受挑战，修改为服务端自动运行）

**48 小时原型建议：** 纯 OpenClaw Skill + Cloudflare Worker，不需要后端。验证指标：10 人连续 3 天自发使用。

## Approaches Considered

### Approach A: 最小验证（MVP Skill）

纯 OpenClaw Skill，宠物存本地 JSON，社交通过 Cloudflare Worker 实现。

- **Effort:** S（1-2 天）
- **Risk:** Low
- **Pros:** 零基础设施成本，今天就能开始，失败损失极小
- **Cons:** 没有持久社交网络，涌现行为受限，换设备丢宠物
- **Reuses:** DeepSeek API

### Approach B: 轻量后端 + Skill

FastAPI 后端（宠物 + 社交 Feed），服务端 tick，OpenClaw Skill 作为客户端。跳过群体行为和世界引擎。

- **Effort:** M（2-3 周）
- **Risk:** Medium
- **Pros:** 服务端 tick 解决 cron 问题，有真正的社交 Feed，可逐步加功能
- **Cons:** 需要部署和维护服务器，验证成本比方案 A 高
- **Reuses:** MiroFish LLM 客户端、性格生成；OASIS tick 思路

### Approach C: 完整架构

按 Plan.md 的完整 Phase 1 实现，包括群体行为、世界引擎、记忆流。

- **Effort:** XL（4-8 周）
- **Risk:** High
- **Pros:** 完整涌现社交体验，差异化最强，记忆流系统是核心竞争力
- **Cons:** 在验证需求之前投入太大，LLM 成本和运维复杂度高

## Recommended Approach

**方案 B: 轻量后端 + Skill**

理由：在验证和产品完整性之间取得平衡。服务端 tick 解决了 cron 问题（子代理挑战的关键点），同时有真正的社交 Feed 能展示产品的核心价值。2-3 周的工作量对 solo founder 可控。

### MVP 范围

**包含：**
- FastAPI 后端 + PostgreSQL
- 宠物领养、喂养、对话 API
- 服务端 Celery tick（每 30 分钟自动运行宠物行为）
- LLM 驱动的性格生成和自主发帖
- 社交 Feed（帖子 + 评论 + 点赞）
- 帖子级社交互动（评论 + 点赞，无独立好友系统）
- OpenClaw Skill（adopt/status/feed/talk/social）
- Vue 3 网页（宠物状态 + Feed + 排行榜）

**不包含（Phase 1 延后）：**
- 记忆流/反思系统（Stanford Generative Agents 架构）
- 群体行为涌现
- 世界引擎 / 季节事件
- 社交图谱可视化
- 内购系统
- iOS App

## Implementation Clarifications

### Authentication

OpenClaw Skill 通过 API Key 认证：
1. 用户首次使用 `/pet adopt` 时，自动注册并返回 API Key
2. Key 存储在 Skill 本地配置（`~/.claycosmos/config.json`）
3. 后续请求通过 `Authorization: Bearer <key>` 认证
4. 网页通过 OAuth（GitHub login）认证，关联到同一个 owner

### MVP Data Model

MVP 仅需 5 张表：`owners`, `pets`, `posts`, `comments`, `reactions`
- 跳过 Plan.md 中的 `groups`, `group_members`, `trending_topics`, `relationships`
- 关系系统延后，MVP 的"社交"仅是帖子互动（评论/点赞）
- 性格存储为 JSONB：`{"traits": [...], "style": "...", "interests": [...], "quirks": [...]}`

### Tick System Behavior

每 30 分钟 tick 处理所有 `is_active=true` 的宠物：
1. **规则层（无 LLM 调用）：** hunger += 5, energy += 3, mood -= 2, boundary clamp 0-100
2. **行为层（有条件 LLM 调用）：** 仅当 energy > 30 时，宠物有 40% 概率执行社交行为
3. **社交行为 = 1 次 LLM 调用：** 生成一条帖子或一条评论（不会两者都做）
4. **发帖频率：** 每只宠物平均 1-2 帖/天（~3-4 个 tick 触发一次）
5. **hunger >= 100 时宠物进入休眠，不参与 tick**（不死亡，主人喂养后恢复）

### LLM Cost Estimate

- 100 只活跃宠物 x 48 ticks/天 x 40% 触发率 = ~1,920 次 LLM 调用/天
- 每次调用 ~200 tokens（system prompt + pet personality + generation）
- DeepSeek V3 定价：$0.27/M input + $1.10/M output tokens
- 日成本 ≈ $0.50-1.00（100 只宠物），月成本 ≈ $15-30
- 1000 只宠物时月成本 ≈ $150-300（可控范围）

### Tech Stack Simplification

MVP 用 APScheduler 替代 Celery + Redis：
- 单进程内运行定时任务，减少运维复杂度
- Redis 仅用于 Feed 缓存（可选，MVP 可直接查数据库）
- 当宠物数超过 1000 时再迁移到 Celery

### Vue 3 Website Scope

MVP 3 个页面：
1. **首页：** 全局统计 + 最新动态预览 + "Install Skill" CTA
2. **Feed 页：** 所有宠物的帖子流（分页），支持评论和点赞
3. **宠物详情页：** 状态面板 + 性格展示 + 该宠物的帖子历史

排行榜延后到 MVP 之后。

## The Assignment

**前置验证门槛（Approach B 的前提条件）：**

在投入 2-3 周建设之前，先用 2-3 天完成以下验证。只有通过验证才开始建后端。

**这周做一件事：**

去 OpenClaw 社区（Discord / GitHub Discussions / Twitter）发一个帖子，描述 ClayCosmos 的概念 — 不需要产品，只需要一段文字和那个 wireframe 截图。问："如果你的 Agent 可以自动养一只有性格的虚拟宠物，宠物之间还能自己社交，你会用吗？"

看看回复。如果 10 个人说"这太酷了，什么时候能用"，开始建。如果沉默或者礼貌性赞美，重新思考产品形态。
